llm-mem

Install once. Keep typing copilot.

A local-first Context Compiler that gives coding tools source-grounded project memory without making users adopt a new daily workflow.

What is it?

Not another coding agent.

llm-mem is an optimization layer. It indexes a repo, remembers durable decisions, compiles task-sized context packs, and exposes them through CLI/MCP surfaces that existing coding tools can use.

CLI local daemon MCP server SQLite index context packs benchmark reports
Why it exists

Agents waste tokens rediscovering the same repo.

Repeated readsSame files, configs, tests, and conventions get reloaded across prompts.
Weak summariesOld summaries go stale, lose citations, or hide important identifiers.
Broad context dumpsWhen the tool cannot prove what matters, users paste too much.
Core idea

Compile context like build systems compile code.

Given a task, repo state, memory, and token budget, llm-mem emits a compact proof-carrying context pack: cited files, constraints, prior decisions, freshness metadata, and expansion handles.

Normal user flow

The daily command should not change.

llm-mem integrate copilot install
copilot

The install step configures project MCP + a lean project skill. After that, Copilot can call llm_mem_context_pack during normal sessions.

What install changes

Small, reversible project files.

.mcp.jsonAdds an llm-mem MCP server entry. Existing MCP servers are preserved.
.github/skills/llm-mem/SKILL.mdAdds clean project guidance without touching org-owned instruction files.
.llm-mem/Stores the local SQLite index, context packs, and benchmark artifacts.
UninstallRemoves only the llm-mem MCP entry, project skill, and optional marked instruction block.
When to use it

Use it when context repeats.

Large reposOnboarding, unfamiliar modules, cross-file fixes.
Long projectsRecurring conventions, decisions, test commands, architecture notes.
Agent-heavy workParallel worktrees, handoffs, repeated investigations.
When not to use it

It is not magic compression.

Tiny one-off scriptsThere may be no repeated context to save.
Fresh empty reposThe index has little signal until work accumulates.
Ambiguous product callsHuman clarification still beats premature context retrieval.
Untrusted contextSource citations matter; uncited summaries should not drive edits.

Benchmark method

A/B against the same task.

baseline-copilotCopilot receives the task prompt only.
llm-mem-contextCopilot receives the task plus a generated, cited context pack. 
llm-mem benchmark run evals\benchmarks\smoke.json --dry-run
llm-mem benchmark run evals\benchmarks\smoke.json --model gpt-5.5
Current status

Working MVP, cautious claims.

Implemented: local indexing, context packs, MCP, Copilot integration, worktree leases, dry/live benchmark harness.

Still needed for strong claims: pinned public fixtures, SWE-bench Lite/Verified subsets, and actual provider token/cost telemetry instead of estimates.

Takeaway

The product is invisible leverage.

Normal users should not manage context manually. They should install llm-mem once, keep using their coding tool, and see the benefit in fewer wasted tokens per successful task.
